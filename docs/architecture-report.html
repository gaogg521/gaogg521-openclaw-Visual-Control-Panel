数据流与状态管理

7.1 整体数据流向

用户操作（点击/搜索/设置） ↓ React 组件（useState + useEffect） ↓ fetch 调用 → Next.js API Route ↓ lib 工具函数（业务逻辑处理） ↓ 多源数据读写： ├── 文件系统：~/.openclaw/openclaw.json、agents/*/sessions/ ├── MySQL：oc_agents、oc_sessions、oc_channels 等 ├── SQLite：agents/*/memory/*.sqlite（同步读） └── OpenClaw CLI：spawn 执行命令 ↓ JSON 响应 → React state 更新 → UI 重渲染

7.2 状态管理策略

策略适用场景实现
本地 useState页面级状态（加载中/错误/搜索/分页）React Hooks
全局 Context主题、语言（跨页面共享）ThemeProvider + I18nProvider
localStorage用户偏好持久化（主题/语言/测试结果）浏览器存储
服务端内存缓存配置数据（30s TTL）config-cache.ts
定时轮询Agent状态(30s)、告警检查(可配置间隔)setInterval
WebSocket 实时Web Chat 实时对话/api/chat/connect

7.3 缓存层次

浏览器层 localStorage（主题、语言、测试结果，永久） ↓ miss API 层 内存缓存（配置数据，30s TTL） ↓ miss 磁盘层 fs.readFileSync（~/.openclaw/*.json） ↓ 异步写入 数据库层 MySQL 连接池（oc_agents/oc_sessions 等）

11.2 关键架构决策

为何选择 Next.js 全栈而不是前后端分离？

  • API Routes 内置，无需单独后端服务，简化部署为单进程
  • standalone 输出模式支持轻量 Docker 镜像（~150MB）
  • 文件路由约定简洁，页面与 API 共享类型定义

为何 MySQL + SQLite 混合存储？

  • MySQL：长期归档、聚合查询、跨 Agent 汇总报表
  • SQLite：Agent 本地高频写入、零配置、无需网络连接
  • 两者互补，文件系统是"真相来源"，MySQL 是"分析层"

像素办公室为何与普通页面完全解耦？

  • Canvas 渲染管道与 React DOM 渲染互不干扰
  • 60fps 游戏循环需要独立 requestAnimationFrame 管理
  • 复杂状态（家具/角色/碰撞）不适合 React state 管理

架构总结

ONE CLAW 龙虾可视化控制面板是一个以 OpenClaw 为核心、以 Next.js 为全栈框架的 AI Agent 运维管理面板。项目最大的特色是将严肃的运维功能（监控/告警/统计/配置）与游戏化可视化（像素办公室）深度融合。

核心亮点

方面设计价值
全栈合一Next.js API Routes 无独立后端单进程部署，运维简单
双数据库MySQL 归档 + SQLite 实时读写特性互补，零依赖本地运行
流式输出SSE 给 CLI 命令长时操作实时进度反馈
游戏化Canvas 引擎与 DOM 解耦60fps 高性能，趣味化运维
多语言/主题6语言 + 5主题全球化支持，高适配性
安全隔离localhost 守卫防止未授权的远程访问
多部署方式Docker / Electron / standalone云端 + 桌面双模支持

✅ 架构优势

  • 模块化清晰：页面/API/工具库三层分离，职责明确
  • 扩展性强：插件式模型/通道/技能支持，新增 Provider 无需改核心
